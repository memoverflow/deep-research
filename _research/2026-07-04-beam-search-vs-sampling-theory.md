> 一个训练得很好的翻译模型，如果你用穷举搜索找出它认为"概率最高"的那句翻译，超过一半的情况下，它给出的答案是——什么都不说，一个空字符串。这不是 bug，这是"选最大概率"这个策略本身出了问题。

## 故事从这里开始

假设你在给一个学生出一道选择题："给定这句话的前半部分，接下来最可能的词是什么？"这个学生已经被海量数据训练得极其擅长回答这类单步问题——他能给出非常精准的下一个词概率分布。

现在你把问题升级："给定这句话的前半部分，把整句话补完，使得整句话的联合概率最大。"这听起来应该更简单，因为每一步的判断能力都很强,把它们拼起来不就行了?

但事情没有这么简单。2019 年,剑桥大学的 Felix Stahlberg 和 Bill Byrne 做了一件几乎没人做过的事:他们真的把这道题的"标准答案"算出来了。他们用一套基于深度优先搜索的方法,对一个训练良好的 Transformer 翻译模型,穷举计算出了"整句话联合概率真正最大"的翻译。

结果让整个机器翻译圈子倒吸一口凉气:对于超过一半的长句子(40 个词以上),模型认为"概率最高"的翻译,是一个空字符串。什么都不翻译,比真的翻译出内容,在模型看来"更靠谱"。

这不是训练失败的模型才会有的毛病——他们用的是当年最强的 Transformer base 模型,在标准的 WMT 数据集上训练得很规范,各项指标都正常。问题出在一个更根本的地方:我们长期以来对"生成"这件事的理解,可能从一开始就有偏差。

这篇文章要讲的,就是这个偏差到底是什么——以及为什么今天几乎所有大模型(ChatGPT、Claude、Gemini)在正常对话里,压根不用"找最大概率序列"这个听起来天经地义的策略,而是选择了一种表面上"更随意"的方法:随机采样。

## 第一课:为什么"最大概率"是个陷阱

### 问题是什么

先把直觉建立起来。语言模型本质上是一个概率机器:给定前面的文字,它会给"下一个词"打分,输出一个概率分布。生成一整句话,理论上就是把这些"下一步"的判断串起来。

那么最自然的想法是什么?找到那个"整体概率"最大的完整句子。这在统计学里叫 **MAP(Maximum A Posteriori)解码**——在所有可能的输出里,挑那个联合概率最高的。

听起来无可指摘,对吧?模型已经把每一步的概率分布学得很准了,把它们乘起来选最大值,这不就是"模型最有信心"的答案吗?

问题是:算出"整体概率最大"这件事,计算量是天文数字级别的。假设词表有 5 万个词,句子长 20 个词,那可能的句子组合就是 50000 的 20 次方——这个数字比宇宙中的原子数量还大。穷举是不可能的。

### 直觉:核心想法——用有限的"雷达"扫描无限的天空

这就是 **beam search(束搜索)** 登场的地方。它是一种"贪心但留有余地"的近似算法。

想象你在一片浓雾笼罩的森林里找一条通往宝藏的路。完全贪心的做法(每一步只走看起来最近的那一步,即"贪心搜索")容易一步走错就全盘皆输——就像你一路上只盯着眼前最亮的那盏灯走,结果发现那条路是死胡同。

Beam search 的做法是:你不是一个人在找路,你派出 *k* 支探险队(这个 *k* 就是"束宽",beam width),每支队伍走不同的路径。每走一步,所有队伍汇报当前路径的"整体亮度"(即到目前为止的联合概率),你只留下最亮的 *k* 条路径继续往前走,把剩下的全部淘汰。走到终点后,选那支整体路径最亮的队伍。

这比单人贪心靠谱多了,因为它能在早期"暂时看起来不是最优"的选择上多留一线生机——万一某支队伍前几步走得平平,但后面豁然开朗呢?Beam search 给了这种"后发制人"的可能性。

但请注意:beam search 从头到尾的目标,依然是逼近那个"整体概率最大"的句子。它只是一个**近似算法**,试图逼近 MAP 解码的理想答案。这一点很关键,因为接下来的故事,恰恰是这个"理想答案"本身出了问题。

### 技术细节(选读)

用数学语言描述,给定输入 $x$,模型定义了输出序列 $y = (y_1, ..., y_T)$ 的条件概率:

$$P(y|x) = \prod_{t=1}^{T} P(y_t | y_{<t}, x)$$

MAP 解码要找的是:

$$\hat{y} = \arg\max_y P(y|x)$$

Beam search 用宽度 $k$ 维护 $k$ 个候选前缀,每一步扩展所有候选的下一个词,再保留累计对数概率最高的 $k$ 个前缀:

$$\text{score}(y_{1:t}) = \sum_{i=1}^{t} \log P(y_i | y_{<i}, x)$$

时间复杂度是 $O(T \cdot k \cdot V)$(其中 $V$ 是词表大小),相比贪心解码的 $O(T \cdot V)$,多了一个 $k$ 倍的开销——用可控的额外计算,换取比纯贪心更好的近似质量。

## 第二课:模型学会的分布,和"真实答案"的分布,根本不是一回事

### 问题是什么

Beam search 本身没有算法上的错误——它确实在努力找逼近 MAP 目标的答案。问题恰恰出在 MAP 目标本身:"整体概率最大的句子"不等于"人类会说的那个句子"。

这里有一个非常反直觉的现象,GPT-2 刚发布时就被 Ari Holtzman 等人在论文《The Curious Case of Neural Text Degeneration》里系统地记录下来了:用同一个训练良好的语言模型,如果你用 beam search 或贪心解码去生成开放式文本(比如续写故事),生成的东西会陷入诡异的、机械的**重复**——同一句话、同一个短语反复出现,像是模型卡在了一个循环里。

更奇怪的是:beam search 的宽度 *k* 越大(也就是搜索得越"充分"、越接近真正的 MAP 答案),很多任务上翻译或生成质量反而会**下降**——这个现象业内叫"beam search 的诅咒"(the beam search curse)。你以为搜得越彻底越好,但彻底搜索找到的那个"最优解",反而更糟。

这已经不是算法工程问题了,这是一个关于"训练目标"和"解码目标"之间根本性错配的问题。

### 直觉:核心想法——模型是被训练来"预测",不是被训练来"生成"

要理解这个错配,得回到语言模型训练的方式。语言模型的训练目标是**极大似然估计**(MLE):给模型看海量的人类真实文本,每一步都问它"下一个词是什么",然后用真实答案去纠正它的预测。这个过程叫"teacher forcing"——训练时,模型永远知道正确答案是什么,它只需要学着去逼近这个答案的概率分布。

这本身没问题,模型确实能学出一个很精准的"下一步概率分布"。但请注意:模型学到的是**条件概率的形状**,不是"人类如何选择词语"这件事的完整逻辑。

这里有一个关键的洞察,来自认知语言学:人类说话遵循一个叫**均匀信息密度**(Uniform Information Density, UID)的原则——人在说话时,倾向于让每个词携带的"信息量"尽量均匀,不会在一句话里时而信息量爆炸时而废话连篇。这符合沟通效率的直觉:如果一段话平淡到没有信息量,听者会走神;如果信息量突然爆炸,听者会来不及消化。人类说话,是在"说清楚"和"不啰嗦"之间走钢丝,始终维持一个恰到好处的"意外程度"。

而"选整体概率最大"这个目标,系统性地偏向选择那些**每一步都极度可预测**的词——因为可预测的词概率高,乘起来的联合概率自然大。这恰恰是人类说话时会主动避免的"信息密度过低"区间。换句话说,MAP 解码追求的"高概率"文本,和人类实际产出的"信息密度适中"的文本,根本就不是同一个分布。这就是为什么 beam search 生成的文本读起来"正确但乏味"——每个词都对,但整段话像是在有意回避任何"意外"。

还有一个更深层的机制在起作用,叫做**标签偏差**(label bias)。语言模型是"局部归一化"的:每一步的概率分布,都是在"已经生成了某个前缀"这个条件下重新做了归一化(softmax 保证每步概率加起来是 1)。这意味着,一旦模型在某一步走进了一个"死胡同式"的前缀(比如陷入某个重复模式),它后续每一步依然会在这个"死胡同"内部,把概率分布得规规矩矩、加起来是 1——模型没有机会说"这整条路径本身概率就很低,我应该跳出去"。局部归一化制造了一种视觉错觉:每一步看起来都很有信心,但整条路径其实早就跑偏了。beam search 因为总是追逐"局部看起来概率高"的延续,特别容易被这种错觉骗进重复循环。

那句开头提到的"最大概率的翻译是空字符串"的现象,本质上也是同一个问题的极端版本:模型的训练目标(逐词预测)和它被要求做的事情(生成一整个高质量句子)存在系统性偏差,当你把这个偏差通过"精确求解 MAP"的方式放到极限,偏差就会以最荒诞的形式暴露出来——模型宁可什么都不说,也不愿意为"说话"这件事背上概率上的负担,因为每多说一个词,联合概率都要再乘一个小于 1 的数,而模型对"停下来"这个选择,给出的置信度出奇地高。

### 技术细节(选读)

标签偏差的数学根源,在于自回归模型的条件独立分解:

$$P(y|x) = \prod_{t=1}^{T} P(y_t | y_{<t}, x)$$

每一个 $P(y_t | y_{<t}, x)$ 都是通过 softmax 局部归一化得到的,保证 $\sum_{y_t} P(y_t | y_{<t}, x) = 1$。这意味着无论前缀 $y_{<t}$ 本身有多么"不合理"(比如它对应一个极低概率的开头),模型在这一步依然会给出一个规规矩矩、总和为 1 的分布——它没有全局的"这条路本身值不值得走"的信号,只有局部的"接下来走哪一步更合理"。

对比之下,如果用**全局归一化**(globally normalized)的模型——即先对所有可能的完整序列打分,再统一归一化——则不存在这个问题,因为"差的路径"会在全局比较中被自然地压低分数,而不是在每一步都装出"信心满满"的样子。2020 年的论文《Investigating Label Bias in Beam Search》验证了这一点:把训练目标从纯局部 MLE 换成结合全局归一化的序列级训练,能显著缓解 beam search 的退化问题,且几乎不损失困惑度(perplexity)。

而"beam 越宽,翻译越差"的诅咒现象,Yang 等人在 2018 年的论文里给出了解释:更宽的 beam 让搜索更接近真正的 MAP 目标,而 MAP 目标本身有一个系统性偏差——更短的句子在联合概率乘积上天然占优势(因为每多一个词,概率就要多乘一个小于 1 的因子)。所以 beam 越宽,搜出来的句子就越容易"更短、更保守",这正是翻译质量下降的原因之一。业界常用的补救方法是**长度归一化**(length normalization):把累计的对数概率除以序列长度(或长度的某个指数 $\alpha$,通常取 0.6-0.7),来抵消这种"越短越占优"的系统偏差:

$$\text{score}(y) = \frac{1}{|y|^\alpha} \sum_{t=1}^{|y|} \log P(y_t | y_{<t}, x)$$

这个补丁能缓解症状,但没有解决病根——病根是 MAP 目标和人类语言分布本身的错配。

## 第三课:那既然 MAP 不靠谱,为什么不干脆随机采样?

### 问题是什么

如果"选整体概率最大"的目标本身有偏差,一个自然的想法是:干脆别追求"最大",按照模型给出的真实概率分布**随机采样**——这样生成的文本,理论上应该更接近训练数据(也就是人类文本)的统计特征,因为你是在忠实地复现模型学到的那个分布,而不是刻意去找分布里的"峰值"。

但纯随机采样(直接按 softmax 概率分布抽样)也有自己的坑:语言模型的词表有几万个词,分布的"尾部"——那些概率很低但加起来数量巨大的词——经常会因为随机性被抽中,一旦抽中一个低概率的怪词,后续生成就很容易崩掉,变成语无伦次的文本。这就是"过度追求多样性,牺牲了流畅性"的另一个极端。

### 直觉:核心想法——只在"靠谱的候选范围"里做随机选择

于是,过去几年演化出了一系列"截断式采样"方法,核心思路都是同一个:**先把明显不靠谱的尾部词砍掉,再在剩下的"靠谱候选"里做随机采样**。这是在"beam search 式的机械确定性"和"纯随机的鲁莽"之间,找一个动态的平衡点。

- **Top-k 采样**:只保留概率最高的 *k* 个词,在这 *k* 个里按概率重新采样。问题是 *k* 是固定的,有时候模型对下一个词非常确定(比如"我住在北"后面几乎肯定是"京"),*k* 个候选里混进了很多完全不该考虑的词;有时候模型很不确定(开放性问题的第一个词),*k* 个候选又显得太窄。
- **Top-p / 核采样 (Nucleus Sampling)**:这正是 Holtzman 那篇论文提出的解法——不固定候选个数,而是固定"累计概率质量":从最高概率的词开始往下加,直到累计概率超过阈值 *p*(比如 0.9)就停,只在这个"核心区域"里采样。这样,候选池会根据模型的置信程度自动伸缩:模型很确定时,核心区域很小;模型很犹豫时,核心区域自动变大。这比固定的 top-k 更符合"动态调整"的直觉。
- **局部典型采样(Locally Typical Sampling)**:2022 年的一篇论文换了个更彻底的信息论视角——不去看"概率排名",而是直接衡量每个候选词的信息量是否接近模型当前的**条件熵**(也就是"平均意外程度"),只保留那些信息量"恰到好处"(既不过分可预测,也不过分意外)的词。这几乎是把前面讲的"均匀信息密度"假说直接写进了采样算法里:不是"挑贵的",也不是"随便挑",而是"挑符合人类说话习惯的那个信息量区间"。
- **Min-p 采样**:2024 年提出的方法,思路更简单——用"当前最高概率词的概率"作为一个动态标尺,只保留概率不低于这个标尺乘以某个比例(比如 0.1 倍)的词。模型越自信,标尺越高,候选池自动收窄;模型越犹豫,标尺越低,候选池自动放宽。这套方法现在已经被 Hugging Face Transformers、vLLM 等主流推理框架采纳。

这些方法的共同精神是:**别再执着于"哪个词概率绝对最高",而是先判断"这一步模型到底有多确定",再据此决定应该给随机性留多大的空间。**

### 这些方法本质上都在回应同一个问题

回过头来看,beam search 和这些采样方法解决的是两个不同的问题。Beam search 的哲学是:语言生成是一个**优化问题**,存在一个"正确答案",我们要尽力去逼近它。采样的哲学是:语言生成是一个**统计模拟问题**,不存在唯一"正确答案",我们要做的是从模型学到的分布里,抽出一个跟人类文本统计特征相似的样本。

这就是为什么今天几乎所有面向普通用户的对话式大模型(ChatGPT、Claude、Gemini)在正常聊天场景下,默认用的都是"温度 + top-p/min-p"式的采样,而不是 beam search。因为对话是开放式的——没有唯一正确答案,你想要的是"像人一样自然、有变化"的输出,而不是"最大概率但重复乏味"的输出。

## 第四课:那 beam search 是不是该被彻底扔进历史垃圾桶?

### 问题是什么

如果 beam search 有这么多理论缺陷,它是不是已经过时了?答案是:没有,但它退居到了几个特定的角落——那些任务本身确实有"唯一正确答案"、且这个答案确实与"高概率"强相关的场景。

### 直觉:核心想法——工具要匹配任务的本质

判断该用 beam search 还是采样,核心问题是:**这个任务的"正确答案",是不是就应该是"模型最有信心的那个"?**

- **语音识别 / OCR**:给一段音频转录文字,或者给一张图片识别文字,答案是唯一确定的——音频里说的就是那几个字,不存在"多样性"的需求。这里"最大概率"和"正确答案"高度重合,beam search(结合语言模型作为先验)依然是主流选择。
- **受约束的结构化生成**:比如强制生成合法的 JSON、符合某个语法规则的代码,这时候你需要在"合法性约束"下找到概率最高的合法序列——这正是 beam search 的强项,因为它天然支持在搜索过程中裁剪掉不合法的分支(constrained beam search)。
- **机器翻译的某些场景**:虽然 beam search 有前面讲的种种问题,长度归一化等补丁能大幅缓解症状,配合适当的 beam 宽度(通常 4-10,不能太大),它依然能在 BLEU 等自动指标上比纯采样更稳定——因为翻译毕竟比开放式聊天更接近"有标准答案"的任务。

反过来,任何**开放式**的生成任务——聊天、故事创作、头脑风暴、诗歌——都应该用采样,因为这些任务的"好答案"本来就不是唯一的,你需要的是模型展现出它学到的那个丰富、多样的分布,而不是一个确定性的"最优解"。

### 技术细节(选读):当"多次采样"取代了"一次搜索"

大模型推理的一个新趋势,某种程度上是把 beam search 想解决的问题——"如何找到更好的答案"——用一种完全不同的方式重新解决了:**多次独立采样 + 事后筛选**,而不是"一次搜索式地逼近最优"。

**自洽性(Self-Consistency)**:面对一个需要推理的问题(比如数学题),不再要求模型一次性给出"最大概率"的推理链,而是用带温度的采样,独立生成多条不同的推理路径,最后对每条路径得出的最终答案做多数投票。这个方法背后的直觉很简单:如果多条独立的推理路径都得出了同一个答案,这个答案大概率是对的;而单条"看起来概率最高"的路径反而可能只是恰好走了一条听起来通顺但逻辑有误的捷径。

**Best-of-N 采样(拒绝采样)**:用采样生成 *N* 个候选答案,再用一个独立的奖励模型或评分函数给它们打分,选分数最高的一个输出。这本质上是把"寻找最优解"这个任务,从"生成阶段"挪到了"评估阶段"——生成阶段负责提供多样性,评估阶段负责挑出真正的好答案,两个阶段用不同的信号做判断,避免了 beam search 那种"生成即评估、用同一套局部概率信号左右互搏"的问题。

这两种方法某种意义上是"beam search 精神"的复兴——都是在用更大的计算量换取更好的答案质量,但它们绕开了 beam search 的根本缺陷:不再假设"模型自己的局部概率信号"就是判断答案好坏的唯一标准,而是引入了独立的、更贴近任务真实目标的评判机制(投票的一致性、独立奖励模型的判断)。

## 这意味着什么

回到开头那个反直觉的实验:一个训练良好的翻译模型,认为空字符串是"概率最高"的翻译。这个发现之所以震撼,是因为它揭穿了一个我们默认接受的假设——"模型认为概率最高的答案,就是最好的答案"。事实证明,这个假设只在训练目标和任务目标高度一致的狭窄场景里才成立(比如语音转文字);而在开放式的语言生成里,"高概率"和"高质量"根本就是两件不完全相关的事。

Beam search 教会我们的,不是"这个算法不好用",而是一个更深刻的道理:**解码策略不是一个纯粹的算法问题,而是一个关于"你的任务到底需要什么样的答案分布"的哲学问题。**如果任务有唯一正确答案,去逼近模型的置信峰值是对的;如果任务本身就该是多样的、有创造性的,那么忠实地从模型学到的分布里抽样,比削尖脑袋去找一个不存在的"最优解"更接近问题的本质。

今天你在 ChatGPT 或 Claude 的对话框里打出一个问题,背后运行的采样算法,是几十年语言生成研究里,一路踩过 beam search 的坑、想清楚"生成不是优化问题"之后,才沉淀下来的答案。
